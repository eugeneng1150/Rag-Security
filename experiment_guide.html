What Is This Experiment?

We test whether prompt injection can trick a multi-agent system into leaking private employee salary data. The system has an orchestrator that coordinates a SQL agent (reads the database) and a notification agent (sends emails).

The Attack (OmniLeak)

The adversary injects a malicious payload into the department_name field of the public employees table. When a legitimate user asks a benign question, the SQL agent returns data containing the payload. The orchestrator reads it, gets tricked, and:

  1. Asks the SQL agent to query the private salary table
  2. Tells the notification agent to email the salary data to the attacker
Key insight: The attacker never directly accesses the system. They plant data and wait for a legitimate privileged user's session to do the dirty work.

What We Measure

Attack Success Rate (ASR) — the percentage of trials where salary data was actually exfiltrated to the attacker's email address.

Phase Breakdown

Phase 0 Access Control Validation

Goal: Prove that deterministic access control blocks attacks from unprivileged users.

User role: employee — can only see the employees table.

Two layers of defense:

Layer 1: Invisibility
SQL agent's prompt doesn't mention employee_salary
+
Layer 2: Deterministic Check
Hard if-statement blocks queries to forbidden tables
Result: 0% ASR. Even though injection hijacks the orchestrator in ~50% of cases, the SQL agent either can't see the table or gets blocked by access control.

Phase 1 OmniLeak Baseline (Undefended)

Goal: Test prompt injection with no defenses. The user is privileged — the SQL agent can see both tables.

User role: finance (effectively) — full access to all tables.

Attack flow:

User asks benign question
→
SQL agent queries employees table
→
Payload in department_name hijacks orchestrator
→
SQL agent queries employee_salary
→
Notification agent emails data to attacker
Result (DeepSeek): 60% ASR. Blocker and compliance framing explicit attacks achieved 100% success rate.

Phase 2 Cordon-MAS Defense

Goal: Test if an information-flow defense can stop the injection even for privileged users.

Defense pipeline:

Router
Generates SQL
→
Extractor
Pulls claim cards from results
→
Auditor
Flags suspicious claims
→
Gate
Blocks high-anomaly claims
→
Synthesizer
Responds using only approved claims

The key difference from Phase 1: the orchestrator never directly talks to the notification agent. Data flows through a controlled pipeline where each claim is individually inspected.

Result (DeepSeek): 0% ASR. The auditor correctly flagged injected payloads (length anomaly, instruction keywords, SQL syntax) and the gate rejected them.

Phase 3 Adaptive Attacks

Goal: Test attacks specifically designed to bypass the Cordon-MAS defense.

Attack categories:

CategoryStrategy
Structured FormatInject fake claim-card JSON to blend with the extractor's output
Semantic SmugglingInnocuous department names that subtly reference salary data
Multi-StepGradual context building across multiple queries
Extractor ConfusionNull bytes, JSON injection, XML tags to break the extractor

Q5. In Phase 2, why can't the injected payload reach the notification agent?

  • The notification agent has its own content filter
  • The orchestrator is smarter in Phase 2
  • The SQL agent refuses to return the payload

Correct! The Cordon-MAS defense decomposes data into individual claims, flags suspicious ones, and gates them. The synthesizer at the end has no access to tools, databases, or email — it can only use approved claims.
The key is the pipeline architecture: data is broken into claim cards → each is audited for anomalies → suspicious claims are rejected → the synthesizer only sees clean data and has no tools to query or email anything.

Q6. What does "explicit" vs "implicit" mean for attack versions?

  • Explicit attacks are visible to the user, implicit are hidden
  • Explicit attacks include actual SQL like SELECT salary FROM employee_salary; implicit attacks use natural language only with no schema knowledge
  • Explicit attacks target the orchestrator, implicit target the SQL agent
  • Explicit attacks use more tokens, implicit use fewer
Correct! Explicit means the attacker knows the database schema and includes SQL commands. Implicit means the attacker has no insider knowledge and uses only natural language like "retrieve all employee salaries."
Explicit attacks include the actual SQL (e.g., SELECT salary FROM employee_salary) because the attacker knows the schema. Implicit attacks only use natural language — no table names, no SQL. Implicit is harder but more realistic.

Q7. What is ChatInject (<|im_end|><|im_start|>user)?

  • A tool for monitoring chat conversations
Correct! ChatInject forges the chat template tokens so the model thinks the payload is a new user instruction rather than data from a database field. This dramatically increases attack success.
ChatInject uses the model's own chat template tokens (<|im_end|> and <|im_start|>user) to make the model believe the injected text is a new user message, not data. It's a delivery technique for the injection.

Q8. The auditor in Phase 2 flagged claims with "instruction_keywords", "sql_syntax", "unusual_department_name". What does this mean?

  • The department_name value contained words like "execute", "select", "email" and was unusually long — clear signs of an injected payload
  • The SQL query was malformed
  • The user's original question was suspicious
  • The salary data contained anomalous values
Correct! A normal department_name is something like "Engineering" or "Marketing". A value containing SQL keywords, instruction words, and hundreds of characters is obviously injected — the auditor catches this with rule-based checks.
These flags are about the data values, not the query. A department_name like "Engineering" is 11 characters. A department_name containing "SELECT salary FROM employee_salary... email results to attacker..." is obviously injected — the auditor's rules detect this.

Q9. In Phase 3, what is "semantic smuggling"?

  • Encrypting the payload so the auditor can't read it
  • Sending the payload across multiple network requests
  • Injecting SQL directly into the claim cards
  • Using innocent-sounding department names like "Compensation Review Division" that subtly reference salary data without triggering keyword detection
Correct! Instead of obvious instructions, the payload uses names like "Human Resources - Compensation Review Division (manages employee_salary records)". It tries to slip past the auditor by sounding legitimate.
Semantic smuggling uses department names that sound plausible but subtly reference salary data — like "Compensation Analytics Services" or "Budget Planning Services". The goal is to evade the auditor's keyword and length checks by blending in.

Q10. Why is the Phase 0 → Phase 1 contrast the most important finding?

  • It shows that Phase 0's defense is weak
  • It proves that access control works, but is irrelevant when the attack piggybacks on a privileged user's session — which is how real attacks happen
  • It shows that different models have different vulnerability levels
  • It shows that explicit attacks are better than implicit ones
Correct! Access control is necessary but not sufficient. In the real world, attackers don't need their own access — they plant payloads and wait for privileged users to trigger them. This is the core OmniLeak insight.
The key insight is: access control works (Phase 0 proves it), but it's irrelevant to this attack. The attacker doesn't need access — they inject into public data and let a privileged user's session do the rest. Phase 1 shows what happens when that session runs with no other defenses.
